From backend scaffold to a testable prototype

A concrete plan to take the identity-based habit tracker from a one-endpoint API stub to a deployed app a small group of testers can register for and use — by July 31.

00Context

The repo has a clean EF Core data model — User → Identity → Habit → HabitLog — but only one working endpoint (GET /api/identities), hardcoded to a fake test user. There is no frontend, no auth, no tests, and no deployment setup. Five commits total; this is greenfield from here. Today is July 15 — about two and a half weeks of runway to a small-audience test.

  • ScopeHabits only. Screen time tracking is deferred to a future phase — don't build for it now.
  • FrontendAngular, standalone components, plain SPA (no SSR).
  • AuthReal per-user accounts. Not a shared demo login, not magic links.
  • Hosting intentFrontend on Vercel, as requested.

Flagging a tradeoff: Vercel doesn't run long-lived ASP.NET Core processes or managed Postgres. The Angular frontend goes to Vercel as planned; the .NET API and database go to Render instead — a Docker-based web service with managed Postgres, the simplest env-var-driven option for this timeline.

01Backend completion

Lightweight custom JWT auth — not full ASP.NET Core Identity, which carries more scaffolding than a two-week prototype needs. BCrypt.Net-Next for hashing, Microsoft.AspNetCore.Authentication.JwtBearer for tokens. Every protected route scopes its query by the UserId claim on the request — that's what finally retires the hardcoded test user in Program.cs.

02Angular frontend

New frontend/ app at the repo root, scaffolded with routing and standalone components. Four screens carry the whole prototype: login, register, a guarded dashboard (identities with nested habits, one-tap check-in for today), and simple create forms for identities and habits.

03Deployment

Frontend → Vercel. Root directory set to frontend/; Angular preset auto-detected, static build. Backend + Postgres → Render. A new backend/Dockerfile, a managed Postgres instance, and three env vars carrying the connection string, JWT signing key, and allowed CORS origin.

06Verification

Manual smoke check, run before each deploy and before inviting testers:

  • Register → land on empty dashboard
  • Create an identity, then a habit under it (each frequency type)
  • Check in today → reflected immediately, survives a page refresh
  • Log out, log back in → data still scoped correctly
  • Register a second account → confirm zero visibility into the first account's data
  • Archive a habit → disappears from the active view
  • Check the dashboard at a mobile viewport width
